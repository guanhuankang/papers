PEARL: Personalized Streaming Video Understanding Model

这篇论文提出了一个新的研究问题 Personalized Streaming Video Understanding (PSVU)， 目标是让视觉语言模型在连续视频流里，边接收用户临时定义的个性化概念，边进行实时与历史问答。 作者同时给出首个对应 benchmark PEARL-Bench，并提出一个无需训练、可插拔的框架 PEARL 作为强基线。整体价值在于：它把“个性化理解”从静态图片和离线短视频推进到更接近真实助手场景的在线视频流。 (Page 1, Abstract; Page 2, Figure 1; Page 3, Introduction)

1. 一句话总结与核心判断

如果把已有个性化 VLM 工作理解为“记住某个人/物，然后在静态图像或离线短视频里回答一次问题”， 那么 PEARL 的贡献就在于把这个范式升级为一个连续、在线、多轮、可动态定义概念的 setting。 论文真正的新意不只是提出一个 framework，而是先定义了一个更贴近实际 AI assistant 场景的新任务 PSVU， 再围绕该任务设计 benchmark 与 retrieval-memory 方案，形成了“任务定义 + 数据集 + baseline framework”的完整闭环。 (Page 1, Abstract; Page 2, Figure 1; Page 3, Introduction)

核心任务 在 streaming video 中动态记住用户定义的 concept，并回答当前或过去时刻的问题。
核心 benchmark PEARL-Bench，覆盖 frame-level 与 video-level personalization，共 132 videos / 2,173 annotations。 (Page 5, Table 2)
核心方法 双粒度记忆系统 + concept-aware retrieval，且 training-free / plug-and-play。 (Page 8, Section 4; Page 9, Figure 3)
核心结果 在 8 个 offline/online baselines 对比中达到 SOTA，并在多种 backbone 上持续增益。 (Page 10-11, Table 3)

2. 研究背景与问题动机

作者指出，现有 personalized VLM 大多停留在静态图片，或者最多是离线视频理解。 这类方法与真实世界的人机交互有明显错位：真实助手往往面对的是连续视频流， 用户会在任意时刻说“这个人叫 A”“这个动作叫 B”，随后马上追问“她现在在做什么”或“他刚刚穿的是什么颜色”。 传统 setting 无法同时满足连续输入、动态概念注册、多轮问答、历史证据检索这几个要求。 (Page 2-3, Introduction; Figure 1)

论文用 customized fitness coaching、social wearable AI、personalized robotics 等应用场景说明， 下一代 personalized AI assistant 需要在视频流上实时响应，而不是在预先裁剪好的短片段上离线推理。 因此，这项工作不只是“把视频变长”，而是在任务范式上从 offline understanding 转向 online streaming understanding。 (Page 2, Figure 1; Page 2-3, Introduction)

3. PSVU 任务定义

PSVU 的核心设定是：输入是一条持续到来的 streaming video，用户可以在任意时间点通过自然语言引入新的 concept， 模型需要把这些 concept 放入 memory，并在后续回答与这些 concept 相关的问题。论文把 concept 分成两类： Frame-level Concepts 与 Video-level Concepts。 前者对应单帧中定义的人/物等静态实体，后者对应跨连续片段定义的动作模式。 (Page 4-5, Section 3.1)

Frame-level Concepts 例如在某一帧定义“这个人叫 <T>”，重点是持续识别该人物/物体。 (Page 4, Section 3.1)
Video-level Concepts 例如定义“这段动作叫 <Action A>”，重点是理解连续动作模式。 (Page 5, Section 3.1)
Concept-Definition QA 用于注册概念，没有最终评分答案，但要求模型正确记住概念。 (Page 5, Section 3.1; Page 6-7, Section 3.3)
Real-Time / Past-Time QA 前者只依赖当前场景，后者必须检索历史证据片段。 (Page 5, Section 3.1)

从形式化定义上，模型需要根据 query 从 evolving concept set 中取出相关 concept subset， 同时从 streaming history 中取出必要 visual context，再联合当前 query 生成答案。 这就把问题自然拆成两个子难点：如何存以及如何取。 PEARL 的设计正是围绕这两个难点展开。 (Page 8, Section 4.1, Eq. 1)

一个很重要的 benchmark 设计点是：它不只看“当前帧能不能识别”，还专门设置了 Past-Time QA 来评估模型是否能够找回历史证据并做时序推理。 对 frame-level 场景，这尤其关键，因为很多问题无法仅凭当前画面回答。 Video-level 场景则更强调动作模式识别，而不是人物外观。 (Page 5-7, Section 3; Figure 2)

为什么这个 benchmark 有意义？
很多现有 benchmark 即使标榜“视频 personalization”，也仍然是短视频、离线、单轮或只支持静态实体。 PEARL-Bench 则把“在线记忆 + 连续概念注册 + 历史检索”一起纳入评估目标，这使它更接近真正的 assistant benchmark。 (Page 5, Table 1; Page 6-7, Figure 2)

6. PEARL 方法详解

PEARL 被设计成一个 training-free、plug-and-play 的框架， 目标不是重训一个全新 video model，而是给现有 VLM 增加 streaming personalization 能力。 整个方法分为两个核心模块：Dual-grained Memory System 与 Concept-aware Retrieval Algorithm。 (Page 8-9, Section 4; Figure 3)

6.1 Dual-grained Memory System

作者把 memory 显式拆成两层：Streaming Memory 负责存视频片段及其 embedding， Concept Memory 负责存用户动态定义的 concept，包括 concept 名称、视觉证据和文本描述。 这种拆法很关键，因为视频流历史与 concept 本身是两种粒度完全不同的信息：一个是时间序列证据库， 一个是个性化词典。 (Page 8-9, Section 4.2; Figure 3a)

  • Streaming Memory：对每个新 clip 计算 multimodal embedding 并存储，用于后续历史检索。(Page 8-9, Section 4.2)
  • Concept Memory：当用户发出 Concept-Definition query 时，抽取视觉证据并生成 compact textual description，再存入记忆。(Page 9, Section 4.2)
  • Frame-level 概念存单帧证据，video-level 概念存整个 clip 作为证据。(Page 9, Section 4.2)

6.2 Concept-aware Retrieval

真实难点不在“记住名字”，而在“用户问名字时，embedding model 其实不认识这个新造词”。 PEARL 的解法是：先在 query 中识别 concept 名称，再用对应 description 把 query 重写成带显式视觉语义的版本， 然后用这个 rewritten query 去检索历史 clip。这个 idea 很实用，因为 retrieval space 更擅长处理 “a young woman with long dark hair” 这类显式描述，而不是“<Y>”这种私有占位符。 (Page 9, Section 4.3; Figure 3b; Appendix C.2)

  1. 从 query 中识别被提到的 concept names，取出对应的 Concept Memory 条目。 (Page 9, Section 4.3)
  2. 将 query 中 concept names 用概念描述替换，得到 rewritten query。 (Page 9, Section 4.3; Appendix C.2)
  3. 对 rewritten query 编码，与所有历史 clips embedding 做 cosine similarity。 (Page 9, Section 4.3)
  4. 选 top-K 相关 clips，并扩展相邻 N 个 clips 获得局部时序上下文。 (Page 9, Section 4.3; Page 13, Hyperparameter Analysis)
  5. 将 concept entries、retrieved historical clips、current clip、original query 一起送入 VLM 输出答案。 (Page 9, Section 4.3)

6.3 Implementation Details

  • Embedding model 使用 Qwen3-VL-Embedding-2B。(Page 10, Section 5.1)
  • Base models 包括 LLaVA-OV-7B、Qwen2-VL-7B、Qwen3-VL-8B。(Page 10, Section 5.1)
  • 默认 retrieval 参数：K = 4；frame-level 取 N = 1，video-level 取 N = 0。(Page 10, Section 5.1)
  • 评估时使用 cyclic option rotation，减少 MCQA 选项偏置。(Page 10, Section 5.1; Page 28, Appendix E.2)

7.4 效率结果

作者强调 PEARL 并不是“高性能但极慢”的方案。以 frame-level avg 为例， LLaVA-OV-7B + PEARL 在 775 ms latency 下取得 38.03， 不仅优于原始 LLaVA-OV-7B，也快于多个 online baselines。 更重要的是，latency 的主瓶颈仍然是底层 LLM inference，而不是 retrieval / rewriting 模块本身。 这说明 PEARL 具备真实在线部署潜力。 (Page 13, Table 5; Figure 5)

8. 论文亮点与启发

  • 亮点 1：任务定义先行。 很多工作直接上方法，而这篇论文先把 PSVU 这个问题形式化，再构建 benchmark 和 baseline，研究闭环较完整。 (Page 3-5, Sections 1 and 3)
  • 亮点 2：显式 concept memory 非常关键。 对 personalization 来说，用户临时命名的 token 本身是无语义的，必须通过描述化、记忆化才能进入 retrieval pipeline。 (Page 9, Section 4.3; Page 12-13, Table 4)
  • 亮点 3：query rewriting 是一个低成本高回报技巧。 它本质上把“私有名字空间”映射回 embedding-friendly 的视觉语义空间。 (Page 9, Section 4.3; Appendix C.2)
  • 亮点 4：benchmark 设计兼顾当前理解与历史回溯。 这比只做实时问答更接近真实 agent / assistant 的长期交互要求。 (Page 5-7, Section 3)

9. 局限性与风险

虽然论文结果很强，但仍有几个值得谨慎看待的点：

  • 数据规模仍不算大。 132 个视频对 benchmark 研究已不错，但相较真实世界 streaming assistant 的复杂度仍有限。 (Page 5, Table 2)
  • Video-level 数据部分来自合成数字人。 这有助于控制动作标签质量，但也可能带来 synthetic-to-real gap。 (Page 6, Section 3.3)
  • 原文对 train/val/test 公开划分未详细描述。 这会影响后续社区复现实验协议的一致性。 (Page 5, Section 3.2; Page 10-13, Section 5)
  • Concept registration 依赖 base model 生成描述质量。 如果 description 过短、过泛或抓错稳定特征，retrieval 质量会受影响。 这一点从 prompt 设计里可以侧面看出。 (Page 9, Section 4.2; Page 22-24, Appendix C.1)
  • 评估仍以多选 QA 为主。 虽然 cyclic option rotation 缓解了选项偏置，但距离开放式真实对话仍有差距。 (Page 10, Section 5.1; Page 28, Appendix E.2)
一个很实际的研究风险：
PEARL 的核心优势来自“名字替换成描述再检索”的机制，因此若未来场景里用户定义的是非常细粒度、 难以用稳定文本描述区分的概念，当前 query rewriting + embedding retrieval 方案可能会遇到瓶颈。 这也是后续可继续扩展的方向。 (Page 9, Section 4.3; Page 22-24, Appendix C)

10. 结论

PEARL 这篇论文最值得关注的地方，在于它把个性化视频理解从“离线、短视频、单轮交互”推进到 “在线、连续、多轮、可动态定义概念”的新范式。PEARL-Bench 为这个问题提供了第一个较系统的评测框架， 而 PEARL 方法则证明：即便不做额外训练，只要把 concept memory、streaming memory 和 retrieval 机制设计好， 现有 VLM 也可以显著增强对 streaming personalization 的支持。 (Page 14, Conclusion)

从研究价值上看，这项工作很像 personalized VLM、streaming video understanding 与 memory-augmented retrieval 三条线索的交叉点。它既是一个 benchmark paper，也是一篇相当有工程可落地性的系统设计论文。
